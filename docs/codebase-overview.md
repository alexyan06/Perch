# Perch — Codebase Overview

_A living summary of what's actually implemented, file by file. Updated incrementally as changes are made — see Change Log at the bottom for history._

_Last updated: 2026-07-14_

## Architecture at a glance

Electron app with two processes, plus a separate Chrome extension:

- **Main process** (`app/src/main/`) — Node.js, has OS access. Owns SQLite, the active-window poller, all three classification tiers, vision escalation, the nudge/mascot state machine, mascot generation + library, mascot window positioning, and a local WebSocket server.
- **Renderer process** (`app/src/renderer/`) — the React UI, sandboxed, no OS/Node access. Two independent windows share this same renderer bundle, distinguished by URL hash: the main window (setup/summary/past-sessions/mascot-library screens) and the mascot window (`#mascot?sessionId=...`).
- **`app/src/preload/index.ts`** — the only bridge between them, exposing a typed `window.api` object to the renderer via `contextBridge`.
- **`app/src/shared/ipc.ts`** — the single source of truth for every IPC payload shape, mirrored in `docs/ipc-contract.md`.
- **`extension/`** — a separate Chrome Manifest V3 extension, talks to the main process over a local WebSocket (not Electron IPC — it's a different program entirely, built and loaded independently).

Data flow for one polling tick: `active-win`/extension signal → `classifier.ts` (tier 1) → if ambiguous, `escalation.ts`'s dwell timer → if still ambiguous past 60s, a screenshot + vision call → written to SQLite (`db.ts`) → pushed to the renderer as `classification:tick` → fed into `nudge.ts`'s stage tracker → `nudge:trigger`/`nudge:clear` pushed to the mascot window, which picks a message from `mascot-messages.ts` and shows it.

## File-by-file — main process (`app/src/main/`)

- **`index.ts`** — app entry point. Creates the main `BrowserWindow` and (per session) the mascot `BrowserWindow`; registers every `ipcMain.handle` for `session:*`, `mascot:*`, and `permissions:*`. `session:start` creates a DB row, starts the poller, creates the mascot window, hides the main window. `session:end` stops the poller, computes stats via `summary.ts`, closes the mascot window, shows the main window, pushes `session:summaryReady`. Note: `SessionActive.tsx` (see below) is only ever mounted in the main window, which is hidden for the entire duration of an active session — in practice it's never actually seen by the user, the mascot window is the real active-session UI.
- **`db.ts`** — the only module that opens SQLite (`better-sqlite3`, singleton `getDb()`). Schema: `sessions`, `classification_events`, `distraction_intervals`, matching PRD §7, with idempotent `ALTER TABLE ... ADD COLUMN` migrations for columns added after the initial schema (`approved_list`, `next_steps`, and the four stats columns). Exposes CRUD for all three tables plus `getSessionStatsSince` (trends aggregation) and `deleteSession` (cascades manually — no `ON DELETE CASCADE` on the child tables, deletes children first inside a transaction). `newId()` (`${Date.now()}-${random}`) is the one ID scheme reused everywhere — sessions, mascots, everything.
- **`poller.ts`** — `startPolling(sessionId, config)` ticks every 7s (fixed interval — PRD's "tighten to 15–30s during an active distraction" isn't implemented, the interval never changes based on classification state). Each tick: `active-win` + `powerMonitor.getSystemIdleTime()`, pulls the latest browser signal from `ws-server.ts` if the focused app is a known browser, runs `classifyTier1`, feeds the result through `escalation.ts`'s tracker, writes a `classification_events` row, pushes `classification:tick`, and feeds the resolved classification into `nudge.ts`'s tracker (pushing `nudge:trigger`/`nudge:clear` as needed).
- **`classifier.ts`** — `classifyTier1(signal, { task, distractionList, approvedList })`: checks distraction list → approved list → task keywords (only words >3 chars count) → else `ambiguous`. Pure function, fully tested (`classifier.test.ts`).
- **`escalation.ts`** — `createEscalationTracker()`: tracks how long the _current_ signal (by `signalKey`, not by classification alone) has been ambiguous; escalates to a screenshot + vision call once that hits 60s continuous. Caches the vision result per signal key so a stable-but-ambiguous window doesn't re-trigger vision every tick. DI-friendly (`captureScreenshot`/`classifyScreenshot`/`getPermissionStatus`/`dwellThresholdMs` all injectable) — fully tested without a real Electron runtime (`escalation.test.ts`).
- **`screenshot.ts`** — `captureActiveWindowScreenshot(windowTitle)`: `desktopCapturer.getSources({types:["window"]})`, matches by exact-or-substring title match, returns base64 PNG or `null`. Lazily imports `electron` inside the function specifically so this module can be loaded in tests without a real Electron runtime.
- **`vision-client.ts`** — the vision API wrapper (per `.codex`'s architecture rule). `classifyScreenshot` uses OpenAI's `gpt-5.6` with a strict JSON schema for vision escalation; end-of-session summaries are computed locally (see `summary.ts` below).
- **`openai-image-client.ts`** — the Gemini image wrapper (the legacy filename remains for compatibility with the existing module imports). `generateBaseSprite`/`generateStageVariant` use Nano Banana Pro (`gemini-3-pro-image`) with the selected photo or prior sprites as references. Prompts explicitly handle three real failure modes found from actual use: multi-subject photos (keep all subjects, apply expression changes to all of them together), non-human subjects (still give them a readable face), and incomplete/patchy fills (explicitly told this is unacceptable).
- **`mascot-image.ts`** — `postProcessSprite`: the deterministic backstop for generated output (nearest-neighbor resize to 64×64, hue-based chroma-key removal of the magenta background, palette quantization) — see `docs/mascot-generation.md` §5 for why this is code and not trusted to the prompt.
- **`mascot-setup.ts`** — in-memory state for the mascot _generation wizard_ only (selected photo, per-stage results as they're generated) — scoped separately from the persisted collection.
- **`mascot-library.ts`** — the persisted mascot collection: `userData/mascots/<id>/{calm,gentle,upset,breakdown,hello}.png` + `metadata.json`, plus `selected.json` for which one's active (`null` = bundled default, a real selectable option). `listMascots`/`selectMascot`/`saveNewMascot`/`deleteMascot`/`getActiveMascotImages`/`migrateLegacyMascotIfNeeded` (one-time migration off the old single-`active/`-directory layout). No DI seam — tests mock the `electron` module itself and point `app.getPath` at a real temp directory (`mascot-library.test.ts`), since this module's whole job is real file I/O.
- **`mascot-position.ts`** — where the mascot window sits on screen, and remembering it. No corner picker anymore — the only default is bottom-right of the primary display. `resolveMascotBounds` (given already-fetched display data and a saved `{x,y}`) is the only pure function left (unit-tested, `mascot-position.test.ts`); `getMascotWindowBounds` is the thin real-`screen`-calling wrapper. Persists to `userData/mascot-position.json` as `{x,y}`, written from `index.ts`'s `win.on("moved", ...)` listener (dragging is native OS window dragging now, not custom-computed — see `Mascot.tsx` below), falling back to the default if a saved position no longer falls on any currently-connected display.
- **`nudge.ts`** — `createNudgeTracker(sessionId)`: the stage 0→1→2→3 state machine. Stage 1 fires instantly on any distraction tick; stage 2 at 20s continuous, stage 3 at 60s; a relapse within 45s of returning to on-task forces an immediate jump to at least stage 2 (two relapses, stage 3). Writes `distraction_intervals` rows on resolve. Fully tested (`nudge.test.ts`).
- **`session-metrics.ts`** — pure aggregation functions, no Electron/DB imports, all tested (`session-metrics.test.ts`): `computeClassificationBreakdown` and `computeCategoryBreakdown` (per-session, walk `classification_events` attributing time between ticks), and `mergeCategoryBreakdowns`/`aggregateSessionStats` (cross-session, used by trends).
- **`summary.ts`** — `generateEndOfSessionSummary(sessionId, endedAt)`: synchronous, no network call. Pulls the session's classification events and runs them through `session-metrics.ts`'s functions. This replaced an earlier prose-summary call entirely.
- **`ws-server.ts`** — the local WebSocket server (port 8743, hardcoded — not actually read from `.env`'s `WS_PORT` despite the env var existing) that the Chrome extension talks to. Handles `connection:hello`/`tab:update`. `getLatestBrowserSignal()` returns the most recent tab update — and, since 2026-07-08, returns `null` again once the socket closes, rather than serving a stale cached signal indefinitely (this was a real bug — see the Change Log and `.codex/skills/activity-classification/SKILL.md`).
- **`permissions.ts`** — macOS Screen Recording (`systemPreferences.getMediaAccessStatus`) and Accessibility (`systemPreferences.isTrustedAccessibilityClient`) status, checked fresh every call. Both `@platform darwin`-gated, always report `true` elsewhere. Also owns the tiny `onboarding.json` "dismissed the permissions gate" flag.

## File-by-file — renderer (`app/src/renderer/src/`)

- **`App.tsx`** — top-level screen router (plain conditional state, no router library). Screens: `setup` | `active` | `past` | `mascotSetup` | `mascotLibrary`, plus a `pendingSummary` state that takes precedence over all of them (shows `SessionSummary` the moment `session:summaryReady` arrives, regardless of what `screen` currently is) and a `permissionsGateNeeded` gate that blocks everything until resolved. Task/distraction/approved-list text state lives here, not in `SessionSetup`, specifically so navigating to the mascot screens and back doesn't lose what was typed.
- **`SessionSetup.tsx`** — task + distraction-list + approved-list inputs (controlled from `App.tsx`, see above), "Start Session", links to past sessions and the mascot library.
- **`SessionActive.tsx`** — elapsed-timer + live classification status, "End Session" button. Fully functional but effectively unreachable in normal use — see the `index.ts` note above.
- **`Mascot.tsx`** — the real active-session UI. Renders either the generated custom sprite or a hand-drawn SVG fallback per stage; subscribes to `nudge:trigger`/`nudge:clear`; shows a transient speech bubble chosen in main from a session-specific message pack (or bundled fallback) that auto-hides after 6s. The 120px character art is a native `-webkit-app-region: drag` surface, while the adjacent X remains a `no-drag` End session button. Main provides its initial side and pushes updates after each native drag based on the midpoint of the display holding the mascot.
- **`shared/mascot-messages.ts`** — the bundled speech-bubble fallback pack and placeholder rendering, used by main and renderer-adjacent summary components. Custom mascot packs are generated in main with the same calm, non-moralizing voice constraints.
- **`SessionSummary.tsx`** — renders `StatusBar` + `CategoryBreakdown` for the session that just ended. No AI-written prose anymore.
- **`StatusBar.tsx`** — the on-task/distracted/ambiguous horizontal stacked bar, status-colored (good/critical/warning), used both for a single session's summary and for the cross-session trends aggregate.
- **`CategoryBreakdown.tsx`** — the ranked time-by-app/site bar chart, single hue (identity comes from the label, not color). Same reuse story as `StatusBar`.
- **`PastSessions.tsx`** — a "Last 7 days" trends card (fetched independently of the session list, via `session:getTrends`) atop the individual session list. Each session card shows the new computed-stats charts, or falls back to the original AI-written prose/next-steps for any session that predates the stats change. Each card has a "Delete" action (`window.confirm` guard).
- **`MascotSetup.tsx`** — the generation wizard: photo → base sprite → review → 3 stage variants generated automatically → save. Unchanged by the library work below; it only ever produces one candidate set at a time.
- **`MascotLibrary.tsx`** — list saved mascots (with the bundled default as a real, always-present option), select, delete (`window.confirm` guard), or jump into `MascotSetup` to create a new one.
- **`PermissionsOnboarding.tsx`** — the first-launch (or whenever permissions are missing) gate: shows Screen Recording + Accessibility status, links to grant each, "Continue anyway" to dismiss.

## File-by-file — extension (`extension/src/`)

- **`background.ts`** — connects to `ws://localhost:8743`, sends `connection:hello` on open, reconnects every 5s on close (matches `docs/ipc-contract.md`'s documented reconnection rule — confirmed correct during the 2026-07-08 investigation below). Sends `tab:update` on `chrome.tabs.onActivated` and `chrome.tabs.onUpdated` (only when a tab finishes loading and is the active one).

## Known gaps vs. the design docs

- **Poller tick rate never tightens.** PRD §6.2 describes tightening the recheck loop to 15–30s once a distraction state is active; `poller.ts` uses a single fixed 7s interval regardless of classification state.
- **No pause/resume toggle.** PRD §8 describes a visible on/off monitoring toggle independent of ending the session; the only control that exists is Start/End.
- **`WS_PORT` is still hardcoded** to `8743` in both `ws-server.ts` and `extension/background.ts`, despite the env var existing in `.env.example` — setting it currently has no effect.
- **The doc-described second vision-escalation trigger isn't built.** `.codex/skills/activity-classification/SKILL.md` describes re-confirming with vision right before a nudge hits stage 2/3 if the last vision check is stale, separate from the dwell-timer trigger — only the dwell-timer path exists in `escalation.ts`.
- **`SessionActive.tsx` is effectively dead UI** in normal usage — see the `index.ts` note above. Not broken, just never actually seen, since the mascot window is the real active-session UI and the main window is hidden the whole time a session is running.
- **`listMascots()` can surface a stale legacy entry.** If migration is skipped because `selected.json` already exists while an untouched legacy `mascots/active/` directory is still present, it gets listed as a phantom mascot with id `"active"` (found while writing `mascot-library.test.ts`; low priority, not fixed).

## Change Log

- **2026-07-04** — Initial version of this doc, written after tracing through `main/index.ts`, `shared/ipc.ts`, `preload/index.ts`, the renderer components, `db.ts`, `poller.ts`, `classifier.ts`, `ws-server.ts`, and `extension/background.ts`. Reflected the codebase at roughly `first-session-tasks.md` steps 1–9 — approved list, tier 2/3, the nudge state machine, and the mascot itself were all still unbuilt at that point.
- **2026-07-08** — Full rewrite. Since the last entry: approved list, tier 2 (dwell timer) and tier 3 (vision escalation), the full nudge/mascot state machine and window, and the past-sessions list all shipped. On top of that: the mascot window now follows across macOS Spaces and is user-repositionable (drag or corner-snap, persisted); a full mascot library replaced the single-mascot slot (save/list/select/delete, with a one-time migration off the old layout); mascot-generation prompts were hardened against three real failure modes (multi-subject photos, non-human subjects, incomplete fills); the end-of-session summary's prose call was removed entirely in favor of locally-computed stats + charts, and the same computation now powers a cross-session "last 7 days" trends view; past sessions and saved mascots can both be deleted; a real bug was found and fixed where a dropped WebSocket connection left the poller feeding vision escalation a stale, non-matching window title (previously misread as a Screen Recording permission problem); and a real test suite now exists (`classifier`, `nudge`, `escalation`, `session-metrics`, `mascot-messages`, `mascot-library`, `mascot-position` — 70 tests, zero at the last entry).
- **2026-07-14** — Reworked mascot dragging end to end after a multi-round debugging saga. Removed the corner-picker/hover-strip entirely — the mascot always starts bottom-right. Tried making the whole visible character draggable via custom pointer-tracking, rewriting the drag math twice along the way: first away from the renderer's `PointerEvent.screenX/screenY` (unreliable on multi-monitor setups with mismatched display scale factors) to `screen.getCursorScreenPoint()` end-to-end; then to make main fully authoritative from the first `pointerdown` (click-vs-drag decided by cursor distance in main, not a renderer-side threshold), which also fixed a real crash (`setPosition` throwing on non-finite coordinates). Along the way, fixed a real, separate macOS-specific bug the same way: `acceptFirstMouse`/`focusable: false` (macOS was swallowing the first click/drag on the inactive mascot window just to activate it). None of this fixed the actual persistent bug — the mascot still couldn't be dragged into the top ~25% of the screen except right at a corner. Diagnosed as macOS Sequoia's window-tiling gesture intercepting the drag there (verified by reading Electron's own native macOS source; also required understanding that WindowServer can distinguish genuine hardware-originated mouse input from synthetic test events for some privileged gesture recognizers, which is why automated real-cursor tests kept passing while real hands-on testing kept failing). Tried `fullscreenable: false` as a targeted exemption — real fix, but insufficient alone (confirmed by hands-on retest). Concluded the custom pointer-tracked drag architecture itself was the wrong approach for this class of OS-level interference and reverted to native OS window dragging (`-webkit-app-region: drag`) instead, which hands the whole gesture to the OS rather than fighting it — this deleted the entire custom drag math (`computeDraggedWindowPosition`/`hasDraggedPastThreshold` in `mascot-position.ts`, the `mascot:dragStart/dragTo/dragEnd` IPC channels) in favor of a small dedicated drag handle (repurposing the existing hover-revealed "Drag to move" hint into a real, functional drag region) separate from the character, since a drag region can't also be a click target.
